Enumerable#Map | The Powerful Cousin of Each

Operating on Collections

February 20, 2016

Introduction to Enumerables

Just when I thought I was starting to get comfortable with the classes and methods I had seen till this week, we were hit with Enumerables! After spending too much time thinking about whether the word "enumerable" sounds funny or not, I did however dive into what it was and came away with many useful, and some quite intuitive lessons.

The Well Grounded Rubyist introduces the concept of Enumerables as a "characteristic of collection objects" that allow for various useful collection behaviors. That part sounded really great, but the book then went on to talk about how classes that use Enumerable "enter into a kind of contract," which made me think of this.

However, thinking of Enumerables in terms of a contract is handy. The class that declares the use of Enumerable has to define an instance method each, and in return, Enumerable makes available the collection class behaviors that sounded so promising. These "behaviors" range from traversal and searching methods, to sorting and mapping methods.

What this means is that when we declare Enumerable as part of an object class (which in turn defines what the method each does for that particular class) we gain the ability to perform various operations to each element in the class one-by-one. Below is an example of how this contract might play out at our local taqueria.

class Taco
include Enumerable
def each
yield "tortilla"
yield "meat"
yield "beans"
yield "salsa"
end
end

Enumerable#map

Of the many "collection behaviors" that become accessible as a result of declaring Enumerable, map is one of the most powerful. In fact, in the week or so that I have been dabbling with Enumerables to solve various coding challenges, I have used map and map! (the in-place version of map) the most.

When map is called on a collection/Enumerable, it returns an array consisting of the results of the operation that was passed to map. Below is an example of how this might work if we were super hungry when ordering a taco.

>> taco = ["tortilla", "meat", "beans", "salsa"]
=> ["tortilla", "meat", "beans", "salsa"]
>> taco.map { |ingredient| "super-" + ingredient }
=> ["super-tortilla", "super-meat", "super-beans", "super-salsa"]

In addition to being a handy fast-food-meal-up-sizer, Enumerabl#map is helpful when coding as it returns a new object that is the result of the intended operation. This is different than each which returns the receiver of the operation. This is a useful way to perform and return operations on any collection object (not just an array as in the above example) so that the results can be returned and/or passed on to additional steps of code.

This also means that map does not alter the received collection (i.e. it is non-destructive). This can be demonstrated by calling our original "taco" array which returns a regular (non-super-sized) taco.

>> taco
=> ["tortilla", "meat", "beans", "salsa"]

In Closing

Enuerables give us the ability to search, sort, and more generally "traverse" the various elements of a collection object, doing something each time. If you are still performing such operations using while and/or for loops with nested control flow logic, check out Enumerables and you will see ways to collapse your code ten-fold. Enjoy!
