Visa Applications | Why Are They Still So Onerous?

Using Ruby Classes

February 27, 2016

Introduction

I had an unplanned family related trip home come up recently, and for reasons only worth getting into over a beer, I had to apply for a Pakistani visa. I have had my fair share of visa application and process nightmares, but this time I stopped and wondered why it was still, in this day and age, such an amazingly onerous process. We have all joked about how Foot Locker manages to maintain all our details on file, but we still have to fill in a from on a clip-board in the ER, and this experience brought on similar thoughts. However, this time I couldn't help also think about how simple this data gathering, crunching, and returning effort could actually be using what we learned this week: Ruby Classes

What is a Ruby Class?

Ruby is an object oriented language. This basically means two things:

  1. All elements (such as arrays, strings, hashes) are objects
  2. All objects are instances of a Class of that object type

The following diagram illustrates this concept

The first example above may seem trivial, but when it comes to classes, it is actually useful to think of a 1.8T model as a specific kind of VolksWagen Jetta, and a VolksWagen Jetta as a specific kind of car. You could take this one step (or any number of steps) further to say that a car is a specific kind of vehicle.

In the same way, an integer is an object of the class "Fixnum," which in turn is a subclass of the class "Integer." (In this example the Integer class is also known as the "superclass" of the Fixnum class) Taking this one step further like we did for the car example, the Integer class is a subclass of the "Numeric" class (which is in turn the superclass of the Integer class)

Based on these principals, the code snippet below can be used to set up a basic visa_application class.

class visa_application
def initialize(application_number)
@application_number = application_number
end
end

The code above is declaring a class of the type "visa_application" and it is initializing it with an application number (a unique number that is associated with each unique visa applications). That all makes sense, but what is the "@applciation_number" all about? The answer is "instance variables."

Instance Variables

Just like in the integer and Jetta examples above, each unique visa application is going to be an instance of the class "visa_application." Object and instance are used somewhat interchangeably and mean the same thing in this context.

The class needs a way to allow all visa applications to have an application number, but also make sure that each unique application (or instance of the class) has its own unique application number that is accessible within that particular instance. This is exactly what is achieved through the instance variable declaration in the code above (instance variables are preceded by the "@" symbol.)

Instance Methods

It's great that our visa_application class now allows us to create new instances of the visa_application class (or simply put, unique visa applications) but that's all it does so far. What if we wanted the class to also generate and manipulate some information pertaining to the particular application in question? This is where instance methods come in to play.

The code below shows how the visa_application class could be supplemented with an "instance method" called "visa_duration". The intent of this method would be to return the visa duration (in this example, the first 2 digits of the visa application number, in months) when called as a method on any instance of the visa_application class.

class visa_application
def initialize(application_number)
@application_number = application_number
end

def visa_duration
@application_number.to_s.slice(0..1)
end

end

Notice how the instance variable @application_number is used inside the visa_duration method. This works because each instance of the visa_application class has access to it's own instance of this application number, and to the subsequent instance methods that are called on it.

Creating Class Instances and Calling Instance Methods

We now have a class that can be used to generate visa applications, and we can call a method on any generated visa application to figure out how long the visa is valid for. Below is the code we can use to implement these two steps.

afaan_visa = visa_application.new(06875467360627)
p afaan_visa.visa_duration
=> 06
# A 6 month visa duration was returned for my visa application number

The power of using classes is that from this point, we could add more and more instance methods to the visa_applciation class to read-in, calculate, create, and return more relevant data about the visa application. This is of course why I am so flustered by how long it is taking for me to get my visa processed and my passport returned to me. The nice thing though is that this blog took a pretty long time too, and it looks like my passport has been delivered since I started writing it!
